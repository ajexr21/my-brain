### Ⅰ. 시장 트렌드 요약: 물리적 미디어 접근 통제의 진화

USB 보안 시장은 **‘암호화의 깊이’**와 **‘운영 환경 의존성 최소화’**를 핵심 목표로 발전하고 있습니다. 사용자가 원하는 '불특정 PC에서 자가 실행하여 암호를 받고 내용을 확인하는' 모델은 현재 시장에서도 매우 유효하며, 그 구현 방식에 따라 기술적 우위가 결정됩니다.

| 구분 | 주력 트렌드 (현재) | 프로젝트 적합성 및 난이도 | 핵심 기술/방향성 |
| :--- | :--- | :--- | :--- |
| **소프트웨어 기반** | *운영체제(OS) 통합 암호화* (BitLocker, FileVault 등). 대용량 데이터를 다루는 기업 환경 표준. | **★★★☆☆ (중급)**: OS의 신뢰도가 필요하므로 '불특정 PC'에 의존할 위험이 있음. 하지만 접근성 면에서 가장 높음. | AES-256 암호화, BitLocker To Go (Portable) 개념 차용. 자가 실행 시 임시 키/패스워드 입력 구조 구현. |

### Ⅱ. 경쟁 제품 분석 (3대 유형 및 강점/약점)

대표적인 세 가지 솔루션 카테고리를 선정하여, 프로젝트의 관점에서 비교했습니다.

#### 1. OS 통합 암호화 솔루션 (예: Microsoft BitLocker To Go)
*   **개요:** 운영체제 레벨에서 USB 전체를 하나의 논리적 디스크로 암호화하고, 부팅 과정이나 접근 시 패스워드를 요구합니다.
*   **✅ 강점 (Strength):** 구현이 비교적 용이하며(OS API 활용), 데이터 손실 방지 및 높은 호환성을 가집니다. 기업 환경에서 가장 흔하게 접하는 방식입니다.
*   **❌ 약점 (Weakness):** **운영체제 의존성**이 높습니다. 최신 버전의 OS나 특정 드라이버가 필요하며, 완벽히 '불특정 PC'에 독립적이라고 보기 어렵습니다.

#### 2. 하드웨어 보안 키/암호화 장치 (예: Kingston IronKey, YubiKey 기반 USB)
*   **개요:** 물리적인 암호화 장치(Dongle)를 사용하며, 이 장치가 마치 '하드웨어 인증서'처럼 작동하여 접근을 허용합니다.
*   **✅ 강점 (Strength):** **물리적 공격 및 원격 해킹에 대한 방어력이 가장 강력**합니다. 소프트웨어의 백도어나 OS 취약점 영향을 받지 않습니다.
*   **❌ 약점 (Weakness):** **매우 높은 비용**이 발생하며, 사용자에게 별도의 물리 장치 구매를 요구합니다. 유연성(flexibility)이 낮습니다.

#### 3. 컨테이너화 및 자가 실행형 소프트웨어 솔루션 (예: VeraCrypt 기반 암호화 볼륨)
*   **개요:** USB 드라이브 내부에 자체적으로 가상의 "암호화된 금고(Container)"를 만들고, 접근 시 별도의 프로그램 또는 스크립트를 통해 비밀번호 인증 후 마운트합니다.
*   **✅ 강점 (Strength):** **비용 효율성이 높고**, OS 의존도를 낮추면서도 강력한 암호화를 구현할 수 있습니다. 프로젝트의 목표(USB 내 자가 실행)와 가장 유사합니다.
*   **❌ 약점 (Weakness):** **사용자 인터페이스(UX)** 설계가 중요합니다. 사용자가 복잡하게 느껴지면 보안 기능을 포기하기 쉽습니다. 또한, 메모리 상에서 키를 처리하는 과정의 안정성 확보가 핵심 기술 문제입니다.

*   **경쟁 우위 강조점:** "OS 비종속성(OS-Agnostic)"과 "최소한의 실행 환경"에 초점을 맞춰, 기기 환경에 관계없이 일관된 보안 경험을 제공함을 핵심으로 내세웁니다. (이는 Researcher가 제시한 하이브리드 트렌드와 완벽히 일치합니다.)

## 📝 I. 핵심 가치 제안 요약 (Value Proposition)
*   **타겟:** 보안 규제가 엄격한 B2B 기업 (법무, 의료, R&D 등).
*   **VP:** 운영체제에 구애받지 않고, 어떤 PC에서도 자가 실행하여 즉각적인 접근 통제를 구현하는 '독립형 게이트웨이'.

## 🔒 II. MVP 필수 기능 목록 (Feature List)

MVP는 반드시 핵심 보안 기능을 **완벽하게 동작**시키는 것에 집중해야 합니다. 추가 부가기능은 모두 제외합니다.

1.  **[Core Security] 자가 실행 메커니즘:**
    *   Windows 환경에서 USB 내부의 특정 프로그램(SecureVault Gateway.exe)을 실행하여 작동.
    *   OS와 독립적으로 동작하는 최소한의 런타임 환경 구축 (메모리 기반 암호화 로직 포함).
2.  **[Access Control] 패스워드 게이트웨이:**
    *   사용자가 접근 시, 강제적인 비밀번호 입력 창을 띄움.
    *   정확한 비밀번호가 입력될 때만 내부 파일 시스템에 대한 읽기/쓰기 권한을 임시로 부여함.
3.  **[Audit Trail] 사용 기록 로깅 (필수):**
    *   누가, 언제, 몇 번 접근했는지(접속 시도 횟수 포함)를 암호화된 로그 파일에 저장하고, 이 로그는 USB 외부에서 읽을 수 없도록 봉인됨. (감사 추적의 핵심).

## 💰 III. 초기 수익화 모델 정의 (Monetization Model V1)
MVP 단계에서는 가장 단순한 **'라이선스 판매 + 연간 유지보수 계약(AMC)'** 형태로 접근합니다.

*   **판매 항목:** SecureVault Gateway 라이선스 패키지 (N개 사용자 기준, 1차 구매 비용).
*   **추가 수익화 요소:** 최소 갱신 시 '운영체제 호환성 업데이트' 및 '최신 암호 알고리즘 적용 유지보수 계약(AMC)'을 필수적으로 제시.

### 🎯 B2B 후속 질문 보강 리스트 (Compliance & Efficiency 초점)

#### 1. 규제 준수 및 감사(Compliance & Audit) 심층 탐색
*   **"만약 외부 감사를 받게 된다고 가정했을 때, 현재 프로세스에서 가장 증빙 자료 확보가 어렵거나 시간이 많이 소요되는 부분은 무엇입니까? (어떤 단계의 '책임자'가 서류 작업을 놓치는 경우가 많습니까?)"**
    *   *(목표: 단순 규정 위반이 아닌, 내부 프로세스의 취약점과 인적 오류를 발견하여 시스템화할 필요성을 부각)*

#### 2. 운영 효율성 및 데이터 흐름(Operational Overhead & Data Flow) 탐색
*   **"A팀에서 B팀으로 업무가 넘어갈 때 (핸드오프 지점), 현재 수기로 작성하거나 여러 시스템을 오가며 발생하는 '정보의 손실'이나 '재확인 시간'이 어느 정도라고 추정하십니까? 이 과정 자체가 비효율성의 근원일 수 있습니다."**
    *   *(목표: 사일로(Silo)화된 업무 프로세스를 파악하고, 통합 플랫폼의 필요성을 강조)*

#### 3. 시스템 확장성 및 미래 성장 제약(Scalability & Future Growth Limits) 탐색
*   **"만약 현재 처리량 대비 연간 트랜잭션 볼륨이 50% 이상 증가한다고 가정했을 때, 지금 사용하시는 방식이나 인력 구조가 감당할 수 있는 최대치와 그 이후에 발생하는 '병목 현상'은 무엇일까요?"**
    *   *(목표: 현재의 Pain Point를 일회성 문제가 아닌, 성장의 장애물(Growth Barrier)로 정의하여 솔루션 도입의 시급성을 높임)*

#### 4. 대체 및 임시 해결책의 숨겨진 비용(Hidden Cost of Workarounds) 탐색
*   **"이 문제를 우회적으로 처리하기 위해 현재 팀원들이 사용하는 '비공식적인 방법(Workaround)'이 있다면 무엇입니까? 그 방법 자체가 유지보수나 시간적 측면에서 발생하는 숨겨진 운영 비용은 얼마 정도일까요?"**
    *   *(목표: 고객 스스로가 해결책의 비효율성을 인지하게 만들고, 우리가 제공할 솔루션의 가치를 '대체하는 노력'에 대한 절약으로 포장)*

#### 5. 성공적인 시스템 도입의 기준(Success Metrics & Adoption Criteria) 탐색
*   **"저희 솔루션을 도입한다고 가정했을 때, 이 투자가 '성공적'이라고 판단할 수 있는 가장 구체적이고 측정 가능한 KPI (Key Performance Indicator)는 무엇일까요? (예: 월 10시간 단축, 규정 위반 리스크 X% 감소 등)"**
    *   *(목표: 추상적인 만족도가 아닌, 고객이 내부적으로 측정하고 인정할 수 있는 정량적 성공 기준을 확보하여 계약 및 도입 근거 마련)*

## 📋 목차 및 기능 정의 (Feature Definition & User Stories)

### F1. 강제 인증 게이트웨이 (Mandatory Authentication Gateway)
**[기능 개요]** 사용자가 물리적 미디어를 연결하거나, 지정된 실행 파일을 실행하는 순간 운영체제의 기본 접근 권한을 우회하고, 반드시 2단계 이상의 인증 절차(PIN/Password)를 강제한다. 이는 모든 데이터 접근 경로에 ‘통제 지점’을 설정하여 인적 실수를 원천 차단한다.
**[비즈니스 목표]** 무단 접근 시도 자체를 기술적으로 봉쇄하고, 관리자에게 통제권(Control Point)을 확보한다.

**▶ 사용자 스토리 (User Stories)**
1. **(일반 직원 관점):** "USB 메모리를 꽂아 아무 파일이나 열려고 할 때, 자동으로 비밀번호 입력창이 뜨게 하여 보안 절차를 거치도록 하고 싶다." *(→ 무단 접근 방지)*
2. **(관리자 관점):** "솔루션 실행 시 반드시 사내에서 지정한 강력한 암호화 정책(예: 12자리, 특수문자 포함)을 강제하도록 설정하고 싶다." *(→ 보안 기준 강제)*
3. **(기술 관점):** "게이트웨이가 작동할 때 OS가 기본적으로 제공하는 '연결 알림'이나 '폴더 열기' 기능을 무력화하여, 사용자에게 우회로를 찾을 여지를 주지 않고 싶다." *(→ 기술적 통제)*

---

### F2. 오프라인 전용 데이터 암호화 및 격리 (Localized Encryption & Isolation)
**[기능 개요]** 보안 폴더에 저장되는 모든 민감 파일과 하위 폴더 구조 자체를 강력한 대칭키 기반으로 암호화한다. 이 기능은 인터넷 연결 여부와 무관하게 작동하며, 접근 권한이 없는 사용자에게는 데이터가 '암호화된 쓰레기'처럼 보이도록 격리되어야 한다.
**[비즈니스 목표]** 기밀 데이터가 물리적으로 유출되더라도(예: USB 분실) 내용물을 해독할 수 없게 만들어 법적 방어 책임을 확보한다.

**▶ 사용자 스토리 (User Stories)**
1. **(직원 관점):** "사내 중요 자료를 담은 USB를 외부에 반출해도, 비밀번호가 없는 사람에게는 접근 자체가 불가능한 암호화된 형태로 보이게 하고 싶다." *(→ 물리적 유출 방지)*
2. **(관리자 관점):** "보안 폴더 내에 저장되는 모든 파일 유형(PDF, DOCX, XLSX 등)을 시스템이 자동으로 식별하고, 해당 유형에 맞는 최신 암호화 표준으로 적용해 주었으면 좋겠다." *(→ 포괄적 보호)*
3. **(기술 관점):** "데이터를 이동하거나 폴더 구조를 복사하는 과정에서도 반드시 암호화/격리 상태가 유지되어, 중간 과정에서 평문 데이터가 노출되는 것을 막고 싶다." *(→ 프로세스 무결성)*

---

### F3. 종합 감사 로깅 모듈 (Comprehensive Audit Logging)
**[기능 개요]** 시스템에 기록되는 모든 활동(접속 시도/성공, 파일 접근, 폴더 변경 등)을 시간순으로 추적하여 중앙 관리자 대시보드에 저장한다. 이 로그는 위변조가 불가능한(Immutable) 방식으로 기록되어야 하며, 규제 준수 보고서 작성의 근거 자료로 활용된다.
**[비즈니스 목표]** 사고 발생 시 '책임 소재'와 '관리 노력(Due Diligence)'을 입증할 수 있는 핵심 증거를 확보한다.

**▶ 사용자 스토리 (User Stories)**
1. **(준법감시 담당자 관점):** "특정 기간 동안 특정 직원이 어떤 파일에 몇 번 접근했는지 리포트를 자동으로 생성하여, 규제 기관에 제출할 준비를 하고 싶다." *(→ 감사 보고서 자동화)*
2. **(관리자 관점):** "로그에서 '비정상적인 패턴'을 감지하고 경고를 받고 싶다. 예를 들어, 평소 접속하지 않던 시간에 대용량 파일이 연속으로 다운로드되는 경우 등이다." *(→ 이상 행위 탐지)*
3. **(개발/IT 관점):** "로그 데이터는 시간과 사용자 ID별로 완벽하게 구분되어 저장되며, 관리자 계정조차도 로그를 임의로 수정하거나 삭제할 수 없게 보호되었으면 좋겠다." *(→ 무결성 보장)*

### F4. 관리자 및 복구 모듈 (Admin & Recovery Module)
**[기능 개요]** IT 관리자가 솔루션 전반을 관리하고, 시스템의 안정성을 유지하는 데 필요한 기능을 제공한다. 여기에는 사용자 계정 정책 설정, 암호화 키 주기적 업데이트가 포함되며, 가장 중요한 '비밀번호 분실 시 복구' 메커니즘이 핵심이다.
**[비즈니스 목표]** 솔루션 도입 후 발생할 수 있는 운영 리스크(정책 변경, 비밀번호 분실)를 최소화하고 안정적인 SaaS 수익 모델을 유지한다.

**▶ 사용자 스토리 (User Stories)**
1. **(IT 관리자 관점):** "새로운 직원이 퇴사하거나 역할을 바꿀 경우, 관리자가 원격으로 해당 계정의 접근 권한을 즉시 비활성화하거나 정책을 변경할 수 있도록 하고 싶다." *(→ 계정 통제 및 거버넌스)*
2. **(사용자/관리자 관점):** "직원이 비밀번호를 잊어버렸을 때, 관리자가 정해진 절차에 따라 안전하게 시스템에 재접속할 수 있는 복구 키 또는 임시 패스워드 발급 기능을 사용하고 싶다." *(→ 운영 연속성)*
3. **(관리자 관점):** "솔루션이 연결된 전체 PC 환경의 보안 정책을 한 번의 대시보드 클릭으로 일괄 적용하거나, 특정 그룹에만 제한적으로 배포할 수 있는 관리 기능이 필요하다." *(→ 중앙 집중식 운영)*

## 🛠️ 3. 개발팀 전달: 핵심 모듈 상세 범위 (Scope Boundaries)

### A. [P1] Audit Log Engine (핵심 로깅 시스템)
*   **범위:** 모든 권한 제한 자산(USB 드라이브, 중요 폴더 등)에 대한 접근 시도 감지 및 기록.
*   **필수 데이터 필드:** `Timestamp`, `User_ID`, `Action_Type` (Read/Write/Delete), `Target_Resource`, `Success_Status`, `Gateway_ID` (솔루션이 구동된 장치 ID).
*   **제약 사항 (Scope Out):** 로그 데이터의 실시간 전송 실패에 대한 복잡한 재시도 로직은 제외. 일단 로컬 저장 후 백업하는 기본적인 구조만 정의한다.

### B. [P2] Basic Dashboard View & Alerting
*   **범위:** P1이 수집한 데이터를 기반으로 3가지 핵심 지표를 실시간 대시보드에 표시.
    1.  최근 N시간 내 실패 접근 시도 횟수 (Failure Count).
    2.  가장 많이 감지된 위험 자산 Top 5 목록.
    3.  전체 모니터링 활성화 상태 여부 (System Health Check).
*   **제약 사항 (Scope Out):** 사용자별 상세 그래프, 추세 분석 등 복잡한 통계 기능은 제외. 단순 카운트 및 리스트 형태로만 구현한다.

### C. [P3] Compliance Report Generator Skeleton
*   **범위:** 관리자 페이지에서 기간(`Start Date` ~ `End Date`)과 사용자를 필터링할 수 있는 UI와 API 엔드포인트 구축. 필터링된 로그 데이터를 **CSV 파일**로 다운로드하는 기능만 구현한다. (PDF 생성을 위한 라이브러리 연동은 다음 단계).
*   **제약 사항 (Scope Out):** 사용자 친화적인 보고서 디자인, 서명/워터마크 삽입 등은 제외. 데이터 추출(Extraction)에 초점을 맞춘다.

from datetime import datetime
from src.models.schema import RawAuditEventRequest, AuditLogEntry

class AuditService:
    """
    AJSoft Secure Vault의 핵심 로직 계층. 
    외부에서 들어오는 원본 이벤트를 검증하고, 불변 로그로 변환하여 저장합니다.
    """
    def __init__(self):
        # 실제 환경에서는 데이터베이스 세션이나 외부 트랜잭션 매니저가 필요함
        print("✅ AuditService 초기화: 로깅 및 무결성 검증 시스템 준비 완료.")

    def _generate_hash(self, data: dict) -> str:
        """주어진 딕셔너리 데이터를 정렬하여 문자열로 변환하고 SHA256 해시를 생성합니다."""
        # 불변성을 보장하기 위해 모든 필드를 순서대로 포함해야 합니다.
        data_string = "".join(str(v) for k, v in sorted(data.items()))
        return hashlib.sha256(data_string.encode('utf-8')).hexdigest()

    def process_raw_event(self, raw_event: RawAuditEventRequest) -> AuditLogEntry:
        """
        1. 원본 이벤트를 받아 내부 로직을 거쳐 최종 AuditLogEntry를 생성합니다.
        2. 데이터 무결성 검증 (Hashing)을 수행하여 로그의 신뢰도를 높입니다.
        """
        print(f"⚙️ [Service] Raw Event Received: {raw_event.user_identifier} - {raw_event.action_type}")
